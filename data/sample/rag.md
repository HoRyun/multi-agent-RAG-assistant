# RAG (Retrieval-Augmented Generation) 개요

RAG는 LLM이 응답을 생성할 때 외부 지식 베이스에서 관련 정보를 먼저 검색해 컨텍스트로 제공하는 패턴이다. 모델의 파라미터에 저장된 지식 외에 최신 정보나 도메인 특화 데이터를 활용할 수 있게 한다.

## RAG가 필요한 이유

LLM은 학습 데이터 기준 시점까지만 알고 있다. 최신 사건, 사내 문서, 개인 데이터에 대한 질문에는 정확하게 답하지 못한다. 또한 모델을 파인튜닝하는 것은 비용이 크고 시간이 걸린다.

RAG는 이 문제를 검색으로 해결한다. 질문과 관련된 문서를 먼저 찾아서 LLM 프롬프트에 포함시키는 방식이다. 모델 가중치를 바꾸지 않아도 새로운 정보를 활용할 수 있다.

## RAG 파이프라인 구성

RAG는 두 단계로 나뉜다.

**인덱싱 단계 (Offline)**
1. 원본 문서를 작은 청크로 분리한다
2. 각 청크를 임베딩 모델로 벡터화한다
3. 벡터를 벡터 데이터베이스에 저장한다

**검색 및 생성 단계 (Online)**
1. 사용자 질문을 벡터로 변환한다
2. 벡터 DB에서 유사도 검색으로 관련 청크를 가져온다
3. 검색된 청크를 컨텍스트로 포함해 LLM에 전달한다
4. LLM이 컨텍스트를 바탕으로 답변을 생성한다

## 청킹 전략

청킹은 RAG 품질에 큰 영향을 미친다. 청크가 너무 크면 관련 없는 내용이 포함되고, 너무 작으면 맥락이 끊긴다.

일반적으로 800~1000자 고정 크기 청킹이 v1 구현에서 무난한 시작점이다. 인접 청크 간 100자 내외의 overlap을 두면 청크 경계에서 맥락이 잘리는 문제를 완화할 수 있다.

## 벡터 유사도 검색

벡터 DB는 임베딩 간 코사인 유사도나 L2 거리를 기준으로 가장 가까운 벡터를 빠르게 찾는다. pgvector는 PostgreSQL에 벡터 연산을 추가한 확장으로, 별도의 벡터 DB 없이 기존 PostgreSQL로 RAG를 구현할 수 있다.

## RAG의 한계

RAG는 검색 품질에 의존한다. 관련 문서가 없거나 청킹이 잘못되면 LLM에 잘못된 컨텍스트가 전달된다. 검색된 청크가 실제로 질문에 도움이 되는지 평가하는 리랭킹 단계를 추가하면 품질을 높일 수 있다.
