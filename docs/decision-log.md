# Decision Log

이 문서는 프로젝트의 주요 기술 결정과 이유를 기록한다.
형식: 선택 / 이유 / 대안 / 포기 이유.

---

## D-001: LangGraph 채택
- **선택**: LangGraph
- **이유**: 상태 기반 멀티 에이전트 흐름을 명시적으로 표현할 수 있음. 학습 목표와 부합.
- **대안**: LangChain 단순 체인, 직접 상태 머신 구현
- **포기 이유**: 단순 체인은 오케스트레이션 학습 효과가 낮음. 직접 구현은 학습 범위 초과.

## D-002: PostgreSQL + pgvector 채택
- **선택**: PostgreSQL 16 + pgvector
- **이유**: 문서 메타데이터와 벡터를 한 DB에서 관리 가능. 기존 경험 재활용.
- **대안**: ChromaDB
- **포기 이유**: ChromaDB는 가볍지만 메타데이터와 벡터를 분리 관리해야 함.

## D-003: Ollama 채택
- **선택**: Ollama
- **이유**: 로컬 실험과 모델 교체가 쉬움. 설치 간편.
- **대안**: vLLM
- **포기 이유**: vLLM은 고성능 서빙 목적이라 학습 단계엔 과함.

## D-004: FastAPI 채택
- **선택**: FastAPI
- **이유**: 기존 경험 재활용. 타입 힌트 기반 검증이 본 프로젝트 코딩 규칙과 부합.
- **대안**: Flask
- **포기 이유**: 비동기 지원과 타입 시스템에서 FastAPI가 현대적.

## D-005: Qwen3 모델 크기 선택 (4B + 8B 이원화)
- **선택**: Qwen3 4B (개발/테스트), Qwen3 8B (최종 실행/데모)
- **이유**: 로컬 환경 스펙(RAM 64GB, VRAM 12GB)을 고려했을 때, 반복 실험에는 4B가 속도 면에서 유리하고, 최종 데모에는 8B로 답변 품질을 높이는 것이 적절함.
- **대안**: Qwen3 4B 단일 사용, Qwen3 14B 이상 사용
- **포기 이유**: 4B 단일은 최종 품질이 아쉽고, 14B는 VRAM 12GB에서 추론 속도가 느려 학습 반복에 부적합.
- **재검토 시점**: 2주차 중반, 4B로 실제 RAG 품질이 충분한지 평가 후 조정 가능.
